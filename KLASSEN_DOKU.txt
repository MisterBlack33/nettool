  CODE RULES
  ─────────────────────────────────────────────────────────────────────────────
  · Klassen: max. ~150 Zeilen
  · Keine Kommentare für selbsterklärenden Code
  · Methodennamen: camelCase, keine Abkürzungen außer std. (ip, os, mac)
  · Keine public Felder außer in Datenklassen (HostResult, LogEntry, …)
  · static factory pattern für Singletons (Holder-Klasse)
  · Alle Exceptions: catch spezifisch, nie Exception/Throwable pauschal
  · System.err nur für wirkliche Fehler, System.out für Info
  · Keine Ausgabe auf System.out aus Tests (nur Assertion-Fehler)

 2  SICHERHEIT & AUTH
════════════════════════════════════════════════════════════════════════════════

UserAuth  (Singleton, ~180 Zeilen)
  PBKDF2WithHmacSHA256, 310.000 Iter., 256-Bit-Key, 32-Byte-Salt
  Benutzernamen lowercase-normalisiert, case-insensitive Duplikat-Prüfung
  Erster Benutzer = Admin.
  createUser / authenticate / changePassword / deleteUser / logout
  isAdmin() / getCurrentRole() / listUsernames() / hasUsers()
  extractStr()  package-private JSON-Helper (auch von AuditLogger genutzt)

AuditLogger  (Singleton, ~120 Zeilen)
  Tab-getrennt: timestamp\tuser\taction\tdetail
  Append-Persistenz über Neustarts. Rotation ab MAX_LINES=200.000.
  log(action) / log(action, detail)
  readRecent(maxLines) / readByUser(username) / clear()
  parse(line) – parst Tab-Format + Legacy JSON-Lines

SecurityMonitor  (Singleton)
  ARP-Spoofing/Poisoning/IP-Spoofing/Rogue-Device-Erkennung
  start(ntfyTopic) / stop() / addBaseline() / addToWhitelist()

LoginDialog
  show(auth) → true bei Erfolg. Merkt loginMonitor für GUI.
